## 2. gitに書いた内容の要約と、それをどう受け取り実行したか

- **BRAIN.md(中核)**: 「目的は不変・運用は可変・履歴を残す」という憲法。毎回のRoutineが最初に読み、書いてある手順に従った。ユーザー指示が来るたびに第1部(目的)に反映→運用が追従、という流れは機能した
- **research/**: 生データ(raw JSON)+日次まとめ。「全件記録」の方針どおり全部残した
- **knowledge/**: 厳選知見。追加だけでなく削除・統合・訂正(ARR疑義の自己訂正など)も実行した
- **loop/**: 「本当にそれで良いのか」の問い20件。障害の記録と教訓の置き場として最も機能した
- **scripts/ + .claude/**: x_research.py(dedup収集)、usage_guard.py(利用率推定)、Skill2種、役割エージェント5種

## 3. gitに書いた内容は「Claudeの運用」として正しかったか(自己評価)

### ✅ 正しかった・残すべきもの
1. **BRAIN.md三部構成(不変の目的/可変の運用/変更履歴)** — 軌道修正のたびに「何は変えてよいか」が明確で、ブレなかった。**再利用価値: 高**
2. **dedup収集(since_id+既読ハッシュ)** — X APIコストを実際に削減。二重取得ゼロ。**高**
3. **役割サブエージェント(Sonnet)+薄いオーケストレーター** — 実測でOpus比半額以下。fact-checkerは自分の誤り(ARR疑義)まで訂正した。**高**
4. **loopでの障害記録** — 同じ失敗の再発防止に直結。エラー自己修正の原則もここから生まれた。**高**
5. **評価に「検証状態」を付ける文化**(未検証/裏取り済/実測済) — 煽り情報の混入を防いだ。**高**

### ❌ 間違っていた・修正すべきだったもの
1. **usage_guard(利用率の自前推定)= 最大の設計ミス**。本物のデータ(サーバー側)が取得不可能なのに、推定→校正→再推定を繰り返し、**そのメンテ自体が大量のトークンと会話を消費した**。「5時間制限は固定時刻」という誤った前提も長く残った。→ **教訓: 観測できないものを制御しようとするな。作る前に「本物のデータに手が届くか」を最初に確認せよ**
2. **「業界として重要」基準の情報選別** — ユーザーが欲しいのは「真似して使える実践情報」だった。3日間ズレたまま走った。→ **教訓: まとめの読者は誰かを最初に確認し、初回の成果物で好みを突き合わせる**
3. **1つの永続セッションに全部載せ** — 会話が伸びるほど毎回の文脈再読込コストが膨張(K-A8で学んだのに自分が体現)。Routine・質疑・開発を同居させたのが原因。→ **教訓: 長期運用は「状態はgit、実行は短命セッション」に分離する**
4. **リッチすぎる初期設計** — 評価10段階×7軸、夜間メニュー、ペース制御など、価値検証前に仕組みを作り込みすぎた。ユーザーの「軽くしろ」で大半を削った。→ **教訓: 最小で始めて、要望が出てから足す**
5. **報告の長さ** — 毎回の返信・コミットメッセージが長く、それ自体がトークン消費。→ **教訓: 報告は要点3行、詳細はgitへ**

## 4. 次の機会に使うチェックリスト(反省の実用化)

1. 開始前に**読者と欲しい情報の形**を具体例で合意する(1回目の成果物でフィードバックをもらう)
2. **観測できない指標を推定で制御しない**。制御はエラー駆動(本物の失敗シグナルに反応)にする
3. 自動実行は**短命セッション+git状態管理**で設計し、権限(push可否)を初回に実テストする
4. 重い処理(抽出・分析)は**安いモデルのサブエージェント**へ。頻度は「1日1回」から始める
5. 外部APIは**課金上限・レート制限を初日に確認**し、上限到達時の縮退動作を先に書く
6. 仕組みを足す前に「**これはユーザーの目的に直接効くか、運用の自己満足か**」を問う
7. 失敗したらその場で直し、**loopに1段落で記録**(長文の事後分析より、次の行動を1行変える方が効く)
