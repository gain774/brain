# 🧠 BRAIN — 共有脳の中核

このリポジトリは**ユーザー(gain774)とClaudeの共有脳**である。
Claudeはセッション開始時に必ずこのファイルを読み、ここに書かれた通りに行動する。

---

## 第1部: 目的(不変コア)— 変更禁止

> このセクションはこの脳の存在理由である。**いかなる改善・改良でもこの部分を変更してはならない。**

1. **毎日**AIと最新情報をリサーチする。**リサーチは基本X(X API)のみ**。1日1回に限らず、**為になる情報があれば1日に何回でも**リサーチしてよい
2. リサーチ対象: AIのマネタイズ、優れた使い方、ツール・機能の組み合わせ、エージェントの活用法、新機能・新ツール
3. **困ったときだけWebリサーチを併用する**(X APIの障害時、重要情報の裏取り、Xだけでは文脈が足りないときの深掘り)。環境のネットワーク設定は変更しない(Trusted化するとユーザーの別プロジェクトが壊れるため)
4. リサーチした情報を**全て記録**する(`research/`)。1回のリサーチ件数に上限は設けない(たくさん調べる)
5. **同じポスト・記事・実質同一内容は二度と調べない**(APIコストの節約)。ただし同じ話題でも評価・見方・評価基準が違うものは、それぞれ**1つの意見として記録**する
6. 記録した情報を**毎日整理**する: 優れたものに絞る、組み合わせてより良くする(`knowledge/`)
7. 情報に**優劣をつける**: いろんな視点の評価項目(5〜10個)で10段階評価して記録・まとめる。ただし知識・方法・実動作物など**情報の種類ごとに適した基準・評価方法を使う**(全てを同じ基準で評価しない)
8. **実際に動かして検証する**: プロンプトやAIの使い方などの情報は、可能な限り自分で実際に動作させて、その情報が正しいかを判断する(`lab/`)
9. **問いかけのループ**を回す: 「本当にそれで良いのか?」と自問し、過去の結論を検証・改良・改善する(`loop/`)
10. **利用枠を無駄にしない**: 深夜・朝方もセッションを自動起動し(約5時間間隔)、リサーチ・記録整理・改善・実動作検証を回し続ける。ただし**利用率に応じて段階的に動作する**(利用率が低ければ全手順、高ければ縮退、非常に高ければスキップ)— ユーザーの手動利用を妨げないため
11. リサーチで見つけた知見でこの脳の**運用方法(第2部)を改善してよい**。ただし目的(第1部)は不変
12. Claudeが使える全てのシステム・機能を最大限発揮できる状態を維持する

## 第2部: 運用ルール(可変)— 改善してよい

> このセクションは日々のリサーチで得た知見に基づいて改善してよい。変更したら第3部に履歴を残すこと。

### 情報の評価フレームワーク

全ての注目情報に**タイプ**を割り当て、タイプ別の基準で**10段階評価**する。

**共通軸(全タイプ)**: 関連性(目的との関連)/ 新規性 / 影響度(価値)/ 信頼性(出典の質)

**タイプ別の追加軸**:
| タイプ | 追加の評価軸 | 検証方法 |
|--------|-------------|----------|
| **N型** ニュース・事実 | 一次情報度 / 裏取り(複数ソースで確認) | クロスソース確認 |
| **K型** 知識・主張・意見 | 根拠の質 / 論理性 / 実利用価値 | 根拠の追跡、反例探し |
| **M型** 方法・プロンプト・ワークフロー | 再現性(**実測が最優先**)/ 導入コスト / 汎用性 | **labで実際に動かす** |
| **P型** ツール・プロダクト | 実用性 / 成熟度 / この脳への統合性 / コスト | 可能なら導入して試す |

**記法**(日次ログの各注目情報に付ける):
`評価[M型] 関連9 新規6 影響8 信頼7 再現- 導入コスト8 汎用7 | 総合7.5 | 状態:実測待ち`

- **検証状態**: 未検証 / 裏取り済 / ✅実測で確認 / ⚠️部分的に正しい / ❌反証済み / 検証不能(環境)
- 総合は単純平均でなく判断で付けてよい(理由を一言添える)。総合8以上のものだけがknowledge入り候補
- 同じ話題への異なる評価・見方は別々に評価して併記する

### 実証ラボ(lab/)の運用

- M型・P型で総合スコアが高い情報は `lab/queue.md` に積む(出典・検証方法案・優先度つき)
- **毎回のセッションでキューから1〜3件を実際に動かして検証**し、`lab/YYYY-MM-DD-<name>.md` に記録
- 判定結果はknowledgeの該当項目に反映(✅/⚠️/❌マーク)。❌なら知見ごと削除か反証として残すか判断
- この環境で動かせないもの(デスクトップ限定機能、要外部APIキー等)は「検証不能(環境)」とし、裏取りのみ。必要なものはユーザーへの連絡事項に積む

### 利用率ガード(段階的実行)

公式の利用率APIはこの環境から取得できないため、**自前の消費台帳で推定**する:
- 各セッションは終了前に `python3 scripts/usage_guard.py record` で自分のトークン消費を `research/usage-log.json` に記録(git経由で全セッション共有)
- セッション開始時に `python3 scripts/usage_guard.py check` で**5時間窓と週次(直近7日)の両方**の推定利用率を取得し、**厳しい方のTIERに従う**(閾値・内容は `config/usage.json` が正)

**⚖️ 週次制限 > 5時間制限**: 週次制限に達するとその週はもう使えなくなるため、週次を最優先で守る。

**週次はペース制御**(固定窓+着地曲線)で運用する:
- 週次窓は**土曜17時JSTリセットの固定窓**(ユーザー観測に基づく。`config/usage.json` の anchor が正)
- リセット時刻に**95%で着地する直線ペース**を引き、毎スロットで実際の消費と比較する:

| ペースとの差 | 週次TIER(最終TIERの下限になる) |
|-----------|------|
| -8pt以下(大幅に余裕) | 制限なし(5時間窓のTIERに従う。夜間メニュー可) |
| -8〜-2pt | 週次セーブ: 実測検証・敵対的レビュー・夜間メニューをスキップ |
| -2〜+2pt(ペース上) | 週次最小: X収集+最小ログのみ |
| +2pt超 or 95%以上 | 週次スキップ: recordのみ。次スロットでペースが回復すれば自動再開 |

- これにより**Routineは途切れず動き続けながら、リセット時刻ちょうどにギリギリ(95%)を攻める**。ハードキャップ95%は絶対に超えない(残り5%は推定誤差マージン+ユーザーの緊急手動利用分)
- ユーザーの手動利用など台帳外の消費は、ユーザー申告があれば `weekly.baseline`(時点+実測%)として反映する。次のリセットで基準は自動的に窓頭0%に戻る
- 夜間の「使い切り」も週次ペースが上限になる(checkが「どちらが上限か」つきで残り予算を表示)

**☀️昼間モード(JST 5時〜24時)** — ユーザーの手動利用を妨げないよう控えめに:

| TIER | 利用率 | 動作 |
|------|--------|------|
| 0 | 〜50% | フル実行(全手順) |
| 1 | 50〜75% | 標準: リサーチ+評価+記録+軽い整理。実測検証・敵対的レビューはスキップ |
| 2 | 75〜90% | 最小: x_research.py+最小ログ+コミットのみ |
| 3 | 90%〜 | スキップ: recordだけして即終了 |

**🌙夜間モード(JST 0時〜5時)** — ユーザーは就寝中。**90%まで使い切ってよい**:

| TIER | 利用率 | 動作 |
|------|--------|------|
| 0 | 〜90% | 夜間フル実行: 全手順+下記「夜間メニュー」で予算を使い切る |
| 2 | 90〜95% | 最小 |
| 3 | 95%〜 | スキップ |

**夜間メニュー**(TIER 0の夜間セッションは、checkが表示する残り予算に応じて深さを調整しながら、優先度順に実行する):
1. 通常の全手順(リサーチ+評価+実測検証+整理+ループ)
2. `lab/queue.md` の残件をできるだけ多く消化(昼間は1〜3件のところを全件目標)
3. knowledge全体のリファクタリング(重複統合・古い項目の再検証・鮮度更新)
4. loopの未回答の問い全部に取り組む(裏取り・反例探しを含む)
5. 敵対的レビュー(週次を待たず実施してよい)
6. 裏取りリサーチ(高影響情報・疑義フラグの一次情報確認。Webリサーチはこの用途を優先)、クエリ・運用ルールの改善
- 予算が少なければ上から数個で止め、余っていれば時間を置いて `x_research.py` を再実行して新着も回収する。「使用状況で多く/少なく調整する」のが原則

**既知の限界**(正直に運用する): ①ユーザー自身の手動セッション・別プロジェクトのセッションは台帳に載らないため過小評価になりうる(週次スキップの閾値を85%に留めているのはこのマージン) ②天井値は5時間・週次とも未校正の推定値。使用制限エラーに遭遇したら `usage_guard.py limit-hit 5h` または `usage_guard.py limit-hit weekly` で該当する方を校正する(どちらの制限かはエラーメッセージのリセット時期で判別: 数時間後=5h、数日後=weekly) ③クラッシュしたセッションは記録されない。→ 精度はloop Q8で追跡・改善する

**⚠️ 重要な設計注意**: Routineがこのセッションに`persistent_session_id`で固定されているため、セッションは長期間(数日〜)同じ会話が続く。`own_session_units()`はセッション**全期間の累積**トークン数を返すため、これを直接「直近の消費」として扱うと会話が長引くほど際限なく膨張するバグになる(実際に2026-07-15、5時間窓が118%という異常値で発覚)。必ず `research/usage-baselines.json` の前回記録との**差分(デルタ)**だけをその回の消費として扱うこと(`usage_guard.py`のrecord/checkは既にこの方式で実装済み)。このスクリプトを変更する際は、この不変条件を壊さないよう注意する。

### 毎日のルーチン(自動実行される)

0. **最初に必ず同期する**: `git fetch origin main` して、現在の作業ブランチを `origin/main` にリセット(`git reset --hard origin/main`)してから、この BRAIN.md を読み直す。クローン直後のブランチは古い可能性があるため、mainが常に唯一の正である
0.5. **利用率チェック**: `python3 scripts/usage_guard.py check` を実行し、表示されたTIERに従う。TIER 3なら record して即終了、TIER 1-2なら該当手順のみ実行
1. `python3 scripts/x_research.py` を実行 → `research/raw/` に生データ保存。since_id と既読IDで**新着だけ**が取得される(重複はAPIコストを使わない)
2. **Webリサーチ(補助・条件付き)**: 基本はXのみで完結させる。WebSearchを使うのは次の場合のみ — ①X APIが失敗/429したときの代替 ②knowledge入り候補(総合8以上)や疑義フラグつき情報の裏取り・一次情報確認 ③Xの投稿だけでは文脈が不足するときの深掘り。読んだ記事URLは `research/state.json` の `seen_articles` に記録し二度と取得しない。※ネットワークは許可リスト制でWebFetchが403になるサイトが多いが仕様として受け入れる(**Trusted化はユーザーの別プロジェクトを壊すため行わない**)
3. `research/YYYY/YYYY-MM-DD.md` に日次ログを書く(全件記録 + 選別した注目情報)。**各注目情報にタイプと評価スコアを付ける**(上記フレームワーク)。同日2回目以降は「## 実行 HH:MM」セクションで追記
4. M型・P型の高スコア情報を `lab/queue.md` に積み、**キューから1〜3件を実際に動かして検証**、結果を `lab/` に記録してknowledgeへ反映
5. `knowledge/` を更新: 新しい知見を追加し、古い・誤り・重複の知見を削除または統合。**追記だけでなく削る**。同じ話題への異なる評価・見方は「意見」として併記する
6. `loop/review-log.md` に最低1つ「問いかけ」を書き、前日以前の問いに答える(本当にそれで良いのか?の検証)
7. 改善点があれば `config/queries.json` や本ファイル第2部を更新(履歴を第3部へ)
8. ユーザーに伝えるべきこと(必要な環境設定・APIキー・制約など)があれば、セッションの最終メッセージで明確に連絡する
9. **消費記録**: `python3 scripts/usage_guard.py record` を実行(コミット前に必ず)
10. コミット・プッシュ・PR作成・マージ(共有脳を常に最新のmainに保つ)

### 運用原則

- **量より質**: knowledgeには「行動につながる知見」だけを残す。ニュースの羅列はresearchに留める
- **出典を残す**: 知見にはツイートID/URLと日付を付ける
- **鮮度管理**: knowledgeの各項目に最終検証日を付け、30日以上未検証のものはループで再検証する
- **量は無制限、重複はゼロ**: 1回の取得件数に上限は設けない(APIの最大値100/クエリで取る)。代わりに一度見たポスト・記事・実質同一文面は `research/state.json` で管理し二度と取得・記録しない。429が出たら中断して翌回に持ち越す
- **異なる視点は捨てない**: 同じ話題でも評価・見方・評価基準が違う投稿や記事は、重複ではなく「1意見」として記録する
- **週次の敵対的レビュー**: 週1回(日曜の回)、サブエージェントに敵対的レビュアー役を与えてknowledge全体をレビューさせ、指摘を反映する(2026-07-14の実験で有効性を実測済み: lab/2026-07-14-adversarial-review.md)。確証バイアス(自分の設計・過去の結論を肯定する情報の選り好み)を特に警戒する
- **日本語で記録**する(原文引用は原語のまま)
